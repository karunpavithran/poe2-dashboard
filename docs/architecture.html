◆ Architecture Decision  ·  PoE2 Arbitrage Dashboard

Normalize at the boundary,
derive at the edge.

The economy API shipped ~588 KB of precomputed, denormalized views, and every new view cost a server change. So I inverted it: the server ships the minimal source — raw exchange edges — and the browser derives every view. New views became a one-file, client-only change.

Outcome

What crosses the wire

same data, three representations

Precomputed views — denormalized cycles + rates, server-built 588 KB
Raw edges — normalized source (edges + value/icon/hub maps) 271 KB  ·  −54%
Raw edges, gzipped — @fastify/compress on the same source 54 KB  ·  −91%

The old payload was 53% arbitrage cycles alone — a 729-cycle explosion over ~1,165 edges, plus another 22% of best-buy / best-sell tables that are cheap to recompute. Shipping the source instead of its expansions is where the win comes from; gzip is just the finish.

Data flow

Source → boundary → derived views

one fetch, cached; every view recomputed in the browser

  1. 01
    Source · poe.ninja

    Undocumented external economy API

    14 exchange categories (Currency, Runes, Essences, …), plus a binary-protobuf builds feed with no published schema.

    • Every category prices against the same Divine / Exalted / Chaos hub
    • Builds facets decoded schema-lessly with pbf + reverse-engineered field numbers
  2. 02
    Server · Poller

    Rate-limited fan-out on an interval

    One poll issues hundreds of throttled requests and runs for minutes — longer than the interval itself.

    • Skip-if-running guard keeps exactly one poll in flight; no starving backlog
    • Snapshot persisted to SQLite → instant render on restart; serve-last-good on error
    • Throttle + retry via fetch-extras, not hand-rolled
  3. 03
    Transport · GET /api/arbitrages

    Ship the source, take no query params

    Returns the full edge set once. No ?categories=, no ?minProfit= — filtering happens client-side over derived rows, so the response is cacheable and identical for everyone.

    • On-demand refresh → 202 immediately; concurrent clicks coalesced
    • isRefreshing / dataAgeMs surface freshness to the UI
  4. ⟶ The boundary edges · currencyValues · currencyIcons · hubs

    Normalize here, derive downstream. This is the only precomputed thing the server owns — the minimal economy source.

  5. 04
    Client · deriveEconomy (react-query transform)

    Rebuild every view from edges — once per fetch, cached

    Runs inside the query function, so components read the same fields they did when the server precomputed them.

    • buildGraph → adjacency map · findTriangularArbitrages → cycles (~47 ms)
    • computeBestExchange → per-hub buy/sell · computeCurrencyCategories
    • Edge-driven & sparse — headroom to move to a web worker if it ever grows
  6. 05
    Views · Exchanges widget

    Arbitrage · Rates · Currency Explorer

    Category chips, profit / volume thresholds, per-hub comparisons — all read from the derived cache.

    • A new derived view is a client-only change: extend deriveEconomy or add a shared function
Domain model

The economy is a directed weighted graph

arbitrage = a cycle whose rate product exceeds 1

Triangular arbitrage, across categories

Exalted → Rune → Divine → Exalted

Divine hub Exalted hub Runes category

Why brute force beats Bellman-Ford here

A few dozen currencies means all ordered triples is at most ~10⁵ checks — cheap, exhaustive, and trivial to reason about. No negative-log-weight transform needed.

for (A,B,C):  rate(A→B) · rate(B→C) · rate(C→A) > 1
  · rotations canonicalized (one cycle, once)
  · both directions kept (rates are asymmetric)

The unlock: every category prices against the same three hubs, so all categories merge into one graph — and profitable cycles route across them.

Honesty rules: observed edges only, never invert a rate to fake the other direction, and route intermediates only through hubs — the path a player would actually take.

chosen to make the edges-first inversion cheap and safe

Framework-free core. The domain layer has one runtime dependency and runs anywhere.

03

packages/shared

The load-bearing layer

No framework. The graph + arbitrage math run identically on server or client — the reason moving the compute across the wire is safe.

  • zod — the only runtime dep; request and response contracts from one source of truth
  • typescript — pure functions, fully typed at the boundary
  • vitest — the derivations unit-tested in isolation, no framework harness
server

Thin, sharp tools

Fetch, normalize, persist, stay fresh — and nothing more. Every non-trivial concern is a tested library, not bespoke code.

  • fastify + @fastify/compress + @fastify/static — schema-first HTTP; compress is the gzip win; static serves the built client in prod
  • drizzle-orm + better-sqlite3 — durable atlas strategies + per-league restart-cache snapshots, layered slice write path
  • fetch-extras — rate-limiting + retry, bought not hand-rolled
  • pbf — schema-less protobuf decode of the builds feed
  • @anthropic-ai/sdk — Claude Haiku 4.5 tags Twitch titles; degrades to untagged on failure
client

Composition over hand-rolling

React 19 + Vite. The query cache doubles as the derived-view store; the UI is assembled from accessible primitives.

  • @tanstack/react-query — the cache layer; deriveEconomy lives in its transform
  • @tanstack/react-table — headless data grids for the rate tables
  • radix-ui + shadcn + tailwind v4 — accessible primitives, not custom widgets
  • downshift · react-router — a11y autocomplete; routing with state persistence

Evolution

How the architecture changed — and why

14 commits · Jul 1 → 11, 2026

The shape wasn't designed up front — it converged. Three recurring judgments drove nearly every structural change, and twice a decision was reversed once it earned a clearer answer.

①Compute at the edge
Push derivation into a portable, framework-free core
②Right-size persistence
Add durability only where the data can honor it
③Fetch on demand
Lazy fetch + snapshot restore over background polling
  1. Jul 1

    v0 — server-computed opportunities

    Baseline

    Server polls poe.ninja, builds the rate graph, brute-forces cycles, and serves computed /api/opportunities?minProfit=&minVolume=. The client is a thin auto-refreshing table.

    WhyShip one end-to-end vertical slice first. The simplest model is "the server does the work, the client renders."

  2. Jul 4

    On-demand poll + URL-persisted UI state

    Added a manual pollOnce; front-end routing with persisted filter state; switched an effect to a ref.

    Why · ③Upstream is rate-limited and slow — blind auto-refresh wastes budget. Hand refresh control to the user and keep view state in the URL.

  3. Jul 7

    The inversion — server → pure snapshot, derive on client

    ★

    A 34-file refactor: moved the graph + arbitrage math out of server/ into the framework-free packages/shared, added bestExchange + currencyCategories, deleted the server-side filters. The server stops computing views and ships raw edges; the client derives every view in the react-query transform.

    Why · ①The payload was 588 KB of denormalized precomputed views — cycles alone were 53%, and each new view cost a schema + poller + server change. Inverting made new views a client-only change and dropped the payload to 271 KB / 54 KB gzipped. (Full decision record below.)

  4. Jul 9

    Persistence, right-sized — SQLite + Drizzle slices

    Revisited

    Introduced Drizzle over better-sqlite3 with a layered slice architecture (router → service → dbFunctions) for atlas and economy. Economy snapshots persist as a restart cache — one upserted row per league, no history; atlas strategies get durable CRUD.

    Why · ②An earlier trade-tracking DB had been built and then deleted — per-trade attribution turned out to be fiction. The database returns only where it earns its place: durable user data and a restart cache, never history the data can't honestly support.

  5. Jul 9

    Deploy groundwork — Docker + CI

    Ops

    A Dockerfile that serves the built client from the same container via @fastify/static, a GitHub Actions workflow, and HOST / CLIENT_DIST wiring. Target: a Raspberry Pi (arm64).

    WhyOne artifact, one container, one small always-on box. No orchestration is warranted for a single-user tool.

  6. Jul 11

    Generalize the pattern — Twitch goes on-demand

    Converted the Twitch slice from a background poller to lazy on-demand + snapshot restore + manual refresh — the same pattern first forged for the economy. Claude tagging now fires only on demand, never on a timer.

    Why · ③Twitch data is stale-tolerant and every fetch spends Twitch and Claude tokens. Don't poll what you can fetch when asked and cache across restarts — a pattern proven on the economy, now reused.

Decision record

The call, structured the way you asked

problem · constraints · options · choice · outcome

Problem

The economy payload was mostly denormalized precomputed views — the cycle list alone was 53% of ~588 KB — and each new view meant a schema + poller + persistence + server change.

Constraints

poe.ninja is rate-limited and a poll takes minutes; the UI must stay responsive; I wanted to iterate on views quickly without a deploy each time.

Options

(a) Keep precomputing server-side; (b) ship normalized edges and derive on the client; (c) ship edges and derive in a web worker.

Choice

Ship the minimal source and derive on the client via framework-free pure functions in a shared package — because the triangular search is sparse (~47 ms) and browser compute is plainly affordable. Worker deferred until measured need.

Outcome

Payload 588 → 271 KB raw → ~54 KB gzipped; new views are one-file client changes; the server's responsibility shrank to fetching, normalizing, and staying fresh.

What makes the inversion safe

Everything past the boundary is a pure function — no framework, no I/O — living in @poe2-dashboard/shared and unit-tested in isolation. The same code runs identically whether the server or the browser calls it, so moving the computation across the wire changed where it runs, never what it returns.
